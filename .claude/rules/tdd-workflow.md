1. 現状調査

- コードベースの構造・既存コード・テスト・規約を確認する

2. Tidying（必要な場合）

- 振る舞いを変えずにコードの構造を整理する
- テストを実行して壊れていないことを確認する
- /security-review を実行する
- コミットする

3. テストを書く（Red）

- 期待する振る舞いをテストとして定義する
- テストを実行して失敗を確認する

4. 実装する（Green）

- テストを通す最小限のコードを書く
- テストを実行して成功を確認する
- /security-review を実行する
- コミットする

5. リファクタリング（Refactor）

- テストが通る状態を保ちながらコードを整理する
- テストを再実行して壊れていないことを確認する
- /security-review を実行する
- コミットする

方針が間違っていた場合

- 正しかったコミットまで git reset で戻す
- ファイルやディレクトリの手動削除は不要

原則

- 小さく書いて、こまめに実行し、ステップごとにコミットする
- 構造の変更（Tidying）と振る舞いの変更は別のコミットに分ける
